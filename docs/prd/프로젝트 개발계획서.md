## 1. PRD: 에이전트 오케스트레이션 및 통합 관리 플랫폼

> **📅 최종 업데이트:** 2026-01-20
>
> **범례:** ✅ 완료 | 🔄 진행 중 | 📋 계획 | ⏸️ 보류

### **제품 비전**

복잡한 소프트웨어 엔지니어링 과업을 자율적으로 분해하고, 전문화된 서브 에이전트들을 통해 실행하며, 사용자가 실시간으로 이를 모니터링하고 제어할 수 있는 고도화된 개발 자동화 환경 구축.

### **주요 사용자 가치**

- **가시성:** 에이전트들이 현재 어떤 사고 과정을 거쳐 작업을 수행하는지 실시간 확인.
    
- **제어권:** 자동화 과정 중 개입하여 방향을 수정하거나 특정 에이전트의 동작을 강제 종료.
    
- **효율성:** 메인 에이전트의 부하를 줄이고 서브 에이전트의 전문성을 극대화하여 정확도 향상.
    

### **핵심 기능 요구사항 (Key Features)**

1. ✅ **계층적 태스크 관리:** 메인 에이전트가 복잡한 Goal을 세부 Task로 분할하고 서브 에이전트에게 할당.

2. ✅ **동적 오케스트레이션:** 작업 결과에 따라 다음 단계를 실시간으로 재설계(Re-planning).

3. ✅ **컨텍스트 격리 및 공유:** 전체 프로젝트 맥락(Global Context)과 개별 작업 맥락(Local Context)의 하이브리드 관리.

4. ✅ **중앙 제어 대시보드:** 에이전트별 상태(Idle, Busy, Error), 토큰 소모량, 작업 진행률 시각화.

5. ✅ **Human-in-the-loop (HITL):** 중요한 결정 지점에서 사용자의 승인을 기다리는 인터럽트 기능.

## 2. TRD: 시스템 아키텍처 및 상세 설계

### **1) 에이전트 역할 분배 및 제어 (Role Distribution)** ✅

- **Main Agent (The Planner):** ✅ `OrchestratorNode`, `PlannerNode` 구현

    - **역할:** 사용자 요구사항 해석, 작업 분해(Decomposition), 서브 에이전트 선정, 최종 결과 검증.

    - **제어 로직:** 상태 머신(State Machine) 기반으로 서브 에이전트들의 생명 주기 관리.

- **Sub-Agent (The Executor):** ✅ `ExecutorNode` 구현

    - **역할:** 특정 도구(Git, Terminal, Editor) 사용, 코드 작성, 테스트 수행.

    - **특징:** 최소 권한 원칙(Principle of Least Privilege)에 따라 필요한 컨텍스트만 주입받음.
        

### **2) 컨텍스트 관리 (Context Management)** ✅

- ✅ **Global Context:** 프로젝트 구조, 주요 기술 스택, 공통 코딩 컨벤션 (Main Agent가 보유).
  - `ProjectContext`, `CLAUDE.md` 기반 구현

- ✅ **Local Context:** 특정 파일 내용, 관련 라이브러리 API, 이전 실행 에러 로그 (Sub-Agent에게 주입).
  - `AgentState.context`, `artifacts` 필드로 관리

- ✅ **Short-term Memory:** 에이전트 간 주고받은 메시지 큐 (Redis 또는 In-memory 활용).
  - `AgentState.messages` + Redis 큐 지원
    

### **3) 오케스트레이션 고도화 (Orchestration)** ✅

- ✅ **ReAct(Reason + Act) 패턴:** 모든 에이전트는 '생각 - 행동 - 관찰'의 루프를 수행.
  - LangGraph StateGraph 기반 구현

- ✅ **Self-Correction:** 서브 에이전트가 실패할 경우, 메인 에이전트가 에러를 분석하여 다른 전략을 가진 서브 에이전트를 재배치하거나 수정을 지시.
  - `SelfCorrectionNode` 구현 (최대 3회 재시도, `error_history` 추적)

- ✅ **Parallel Execution:** 의존성이 없는 태스크는 병렬로 처리하여 전체 리드 타임 단축.
  - `ParallelExecutorNode` 구현 (`asyncio.gather`, 최대 3개 동시 실행)
    

### **4) 대시보드 및 모니터링 (Monitoring)** ✅

|**구성 요소**|**상세 내용**|**상태**|
|---|---|---|
|**Task Tree View**|메인 태스크부터 하위 태스크까지의 트리 구조와 진행 상태(%) 시각화|✅ `TasksPage.tsx`, `TaskPanel.tsx`|
|**Agent Logs**|개별 에이전트의 추론 과정(Chain of Thought) 실시간 스트리밍|✅ `ActivityPage.tsx`, WebSocket|
|**Resource Monitor**|LLM API 호출 횟수, 사용 토큰량, 예상 비용 계산|✅ `CostMonitor.tsx`, `TokenUsage` 추적|
|**Control Panel**|특정 에이전트 Pause/Resume, 특정 단계 재시도(Retry) 버튼|✅ Cancel/Delete 구현, 📋 Pause 계획|

# <font color="#00b0f0">프론트엔드 상태 관리 전략</font> ✅


## 1. 에이전트 통신 프로토콜 (JSON Message Schema) ✅

에이전트 간의 대화와 오케스트레이터의 제어를 표준화하기 위한 메시지 규격입니다. 모든 메시지는 `TraceID`를 공유하여 태스크의 기원을 추적합니다.

> ✅ **구현 완료**: WebSocket 메시지 스키마 구현 (`handleMessage` in `orchestration.ts`)

JSON

```
{
  "header": {
    "messageId": "msg_12345",
    "traceId": "trace_abcde", // 전체 태스크 흐름 추적용
    "timestamp": "2026-01-11T11:27:56Z",
    "sender": "Main-Orchestrator",
    "receiver": "Sub-Agent-Coder-01"
  },
  "payload": {
    "type": "TASK_ASSIGN", // TASK_ASSIGN, THOUGHT, ACTION, OBSERVATION, ERROR, APPROVAL_REQ
    "status": "pending",
    "content": {
      "task": "src/components/Button.tsx의 리팩토링",
      "context": ["Ref: style-guide.md", "Current Code: ..."],
      "tools_allowed": ["read_file", "write_file", "terminal_run"]
    }
  },
  "metadata": {
    "model": "claude-3-5-sonnet",
    "tokenUsage": { "prompt": 1024, "completion": 512 }
  }
}
```

### **핵심 메시지 타입**

- **THOUGHT:** 에이전트의 내부 추론 과정 (Chain-of-Thought).
    
- **ACTION:** 도구 호출 명령 (예: `npm test` 실행).
    
- **OBSERVATION:** 도구 실행 결과 (예: 테스트 패스 여부).
    
- **APPROVAL_REQ:** 사용자 승인이 필요한 위험 작업 알림.

## 2. 프론트엔드 상태 관리 전략 (Streaming & State) ✅

에이전트 데이터는 실시간성(Streaming)이 강하고 양이 방대하므로, 일반적인 REST API 방식과는 다른 접근이 필요합니다.

> ✅ **구현 완료**: Zustand + WebSocket 기반 실시간 상태 관리

### **데이터 흐름 아키텍처**

1. ✅ **Transport:** **WebSocket** 또는 **Server-Sent Events(SSE)**를 통해 에이전트의 출력을 실시간 스트리밍합니다.
  - WebSocket: 양방향 통신 (태스크 제출, 승인 응답)
  - SSE: 단방향 스트리밍 (Claude Sessions 모니터링)

2. ✅ **Store (Zustand ~~또는 Redux~~ 추천):**

    - **Global Store:** 전체 태스크 트리와 에이전트 목록 관리.
      - `orchestration.ts`, `claudeSessions.ts`, `navigation.ts`

    - **Log Buffer:** 실시간 로그는 너무 잦은 렌더링을 방지하기 위해 100ms~200ms 단위로 **배치(Batch) 업데이트**를 수행합니다.

3. 📋 **Cache (React Query):** 에이전트 설정값, 프로젝트 구조 등 정적인 메타데이터 관리.
  - 현재: Zustand persist 사용


### **최적화 기법: Virtualized Logs** 📋

에이전트의 로그가 수만 줄이 넘어갈 경우 브라우저 성능이 저하됩니다. **`react-window`** 또는 **`virtuoso`**를 사용하여 화면에 보이는 부분만 렌더링하는 가상 리스트(Virtual List) 적용이 필수적입니다.

## 3. 오케스트레이션 제어 시퀀스 (Execution Flow) ✅

메인 에이전트와 서브 에이전트가 어떻게 협업하는지 보여주는 워크플로우입니다.

> ✅ **구현 완료**: LangGraph StateGraph 기반 워크플로우

```
    plan: List[str]
    current_step: int
    worker_logs: List[dict]
    final_result: str

# 2. 메인 에이전트: 계획 수립 (Orchestrator)
def orchestrator_plan(state: AgentState):
    llm = ChatAnthropic(model="claude-3-5-sonnet-20240620")
    # 태스크를 세부 단계로 분할하는 로직

# 3. 서브 에이전트: 실행 (Worker)
def worker_execute(state: AgentState):

    step_to_run = state['plan'][state['current_step']]
```

Python

```
class CircuitBreaker:
    def __init__(self, max_tokens=50000, max_iterations=10):
        self.total_tokens = 0
        self.iterations = 0
        self.max_tokens = max_tokens
        self.max_iterations = max_iterations

    def check_and_update(self, usage: int):
        self.total_tokens += usage
        self.iterations += 1
        
        if self.total_tokens > self.max_tokens:
            return False, "Budget Exceeded"
        if self.iterations > self.max_iterations:
            return False, "Max Iterations Reached"
        return True, "OK"
```

에이전트가 로컬 환경이나 서버의 코드를 직접 수정할 때는 **보안**과 **파일 무결성**이 가장 중요합니다.

### **파일 조작 프로토콜**

- **Virtual File System (VFS):** 에이전트가 직접 파일을 수정하기 전, 가상 메모리 상에 파일 트리를 복제하여 수정을 시뮬레이션합니다.
    
- **Atomic Write:** 모든 수정은 `Temporary File`에 먼저 기록된 후, 테스트(Lint, Build)를 통과했을 때만 `Commit/Overwrite`됩니다.
    

### **샌드박스 제어 코드 (Node.js/Python 예시)** ✅

> ✅ Docker 컨테이너 기반 샌드박스 구현:
> - `network_mode: none` - 네트워크 격리
> - `mem_limit: 512m` - 메모리 제한
> - `user: sandbox` - non-root 실행
> - `security_opt: no-new-privileges` - 권한 상승 차단

에이전트에게 허용된 경로 외에는 접근하지 못하도록 `chroot`나 `Docker Container` 환경에서 실행하는 것이 권장됩니다.

Python

```
class FileSandbox:
    def __init__(self, root_dir: str):
        self.root_dir = os.path.abspath(root_dir)

    def safe_write(self, relative_path: str, content: str):
        full_path = os.path.abspath(os.path.join(self.root_dir, relative_path))
        if not full_path.startswith(self.root_dir):
            raise PermissionError("Access denied: Outside of sandbox")
        
        with open(full_path, 'w') as f:
            f.write(content)
```

### **8.1 병렬 작업 관리 (Parallel Task Management)** ✅

| 기능 | 설명 | 구현 방식 | 상태 |
|------|------|-----------|------|
| **Concurrent Execution** | 의존성 없는 태스크 동시 실행 | `asyncio.gather()` + Semaphore | ✅ |
| **Batch Processing** | 2개 이상 ready 상태 태스크 자동 배치 | OrchestratorNode 자동 감지 | ✅ |
| **Resource Pooling** | 최대 동시 실행 개수 제한 (기본: 3) | 설정 가능한 Semaphore | ✅ |

```python
# orchestrator/parallel_executor.py
class ParallelExecutorNode(BaseNode):
    async def run(self, state: AgentState) -> dict:
        semaphore = asyncio.Semaphore(MAX_CONCURRENT_TASKS)
        batch_ids = state.get("batch_task_ids", [])
        results = await asyncio.gather(
            *[self._execute_with_semaphore(tid, semaphore) for tid in batch_ids]
        )
        return {"tasks": merge_results(results)}
```

```python
class LeadOrchestratorAgent(BaseAgent):
    async def execute(self, task: str, context: dict) -> AgentResult:
        # 1. 태스크 분석 (LLM 기반)
        analysis = await self._analyze_task(task, context)

        # 2. 에이전트 할당 검증
        subtasks = self._validate_agent_assignments(analysis.subtasks)

        # 3. 실행 계획 생성
        execution_plan = self._create_execution_plan(analysis)

        return AgentResult(success=True, output={
            "type": "orchestration_plan",
            "analysis": analysis,
            "execution_plan": execution_plan,
        })
```

```python
class LeadOrchestrator:
    async def orchestrate(self, goal: str, context: dict) -> WorkflowResult:
        # 1. 태스크 분해
        subtasks = await self._decompose_goal(goal)

        # 2. 의존성 분석 및 병렬화 가능 태스크 식별
```

```python
    reason: FeedbackReason | None  # incorrect, incomplete, off_topic, style, performance, other
    reason_detail: str | None
    original_output: str
    corrected_output: str | None  # implicit 피드백 시

class DatasetEntry(BaseModel):
    id: str
    feedback_id: str
    system_prompt: str
    user_input: str
    assistant_output: str
    is_positive: bool
    metadata: dict[str, Any]
    created_at: datetime
```
